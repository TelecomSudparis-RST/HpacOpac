vCDN Infrastructure Optimization Simulator

Use this web page to operate the Optimization on the Infrastructure using HMAC Algorithm
Use the Admin page for manual alteration of Infrastructure parameters and values

About


This is a demo implementation of HMAC algorithm for Optimal vCDN Migration on an OpenStack environment.
HMAC is an Optimal vCDN migration algorithm that uses topologie information, infrastructure metrics and the demands for a vCDN service as inputs. These input information can come from Operator's systems or, in our case, from manual input. The objective of HMAC is to determine where to migrate a VM in order not to saturate the network links capacity neither the OpenStack capacity, minimizing the migration cost. This is an heuritic algorithm, it not an exact sollution.
For the Virtualization Infrastructure values (OpenStack), this demo gathers life statistics and metrics directly from OpenStack Controller and Telemetry services. For VM Migration over OpenStacks DataCenters (each DC having its Controller/Network/Telemetry services), this demo integrates with OpenStack KeyStone, Nova, Glance and Ceilometer via API calls. After the Simulation stage, some actual Instantiations and Migrations can be triggered from vIOS, affecting the OpenStack VMs. For DNS Integration (optional), this demo uses a single OpenStack Designate Server holding the DNS Zones to be modified when migrating VMs from an OpenStack DC to another (this is needed because each OpenStack DC can have its own public addressing space from where the Floating IPs are assigned).
Additionally, this simulator uses an exponential algorithm that relates the QualityOfExperience metric perceived by the users (in MOS values) with the BW available at the Infrastructure. This way, it can simulate a reduction in the vCDN traffic BW able to sustain a desired QoE metric for the clients in the infrastructure. This would mean that the Infrastructure operator can simulate a BW reduction that still respects the QoE SLA agreed.

Concepts

  • Topologie: The network topologie is described by the Locations where the infrastructure has an installation, and by the Network Links that connect those locations, having a certain link BW capacity. ClientGroups is an aggregate of clients with common characteristics that are associated to a Location where they connect to the infrastructure.
    The Topologie is shown as a Graph and a Gomory-Hu Tree.
    In this Demo, the Network Links can be generated randomly, from a random graph with a random capacity, and the ClientGroups are placed in the borders of the random topologie.
  • vCDN: The client for the operator's infrastructure is a CDN provider, identified by an OpenStack Tenant/Project credentials. This tenant/project can login on one or more of the POPs. The vCDN also identifies the amount of resources it would deploy when instantiated in a POP.
    The assemble of servers and resources used by a vCDN in a POP is called an Instance. The sum of all the needed resources is indicated for each vCDN.
    The vCDN defines the average QoS BW given to the ClientGroup requests.
  • POP: The infrastructure has Points of Presence where an Open Stack DC is deployed, in a certain Location.
    The POP is identified by the OpenStack Keystone Public endpoint and admin credentials, along with the installed capacity (might defer from the values obtained directly via the OpenStack API)
    The Hypervisors of all the POPs are queried and the sum of hypervisors' capacities is the total POP capacity
    A POP has a quota of resources and can hold a set of Servers (VMs) for a vCDN. The assemble of Servers and resources used by a vCDN in a POP is called an Instance
  • Demand: Periodically either the vCDN or the operator feeds HMAC with the existing Demands, the requests for service.
    A Demand comes from a ClientGroup and targets a vCDN in a specific POP. This is to relate to the fact that CDN internals take care of client request redirection to the appropriate cache server.
    A Demand has a Volume property (could be the number of concurrent connections, the number of devices) that gives an idea of the magnitude of the Demand.
    In this Demo, the Demands can be generated randomly, requesting services from all the Instances with a random volume
  • Migrations: As a result from the HMAC optimization, for a set of Demands presented, there is possibly a Migration suggested.
    All suggested Migrations have a Cost and Delay associated, and a minimum BW available in the migration path. The list of Migrations can be sorted on any of these parameters.
    The Cost is (vCDN Disk size) * (migration path hops ) * (Migration Cost Multiplier) . This is to have take into account the size of the data to be migrated and the distance of the migration.
    The Migration Cost Multiplier is set as a positive value that multiplies the cost of a migration between any 2 POPs. It is bidirectional.
    The Migration is only possible if the other Demands that are requesting the vCDN from the original POP do not cause another saturation when they are redirected to the new migrated POP.
  • Redirections: As a result from the HMAC optimization, it could be detected that the Demands requesting a vCDN in a POP could be redirected to the another POP where the same vCDN is already instantiated.
    There is no actual Migration in this case, but the Demands could be redirected to the optimal POP.
  • Instantiations: As a result from the HMAC optimization, it could be detected that the Demands request a vCDN in a POP where it is no instantiated yet instantiated.
    There is no actual Migration in this case, but the vCDN could be instantiated in the optimal POP.

Optimization Decision


The Decision to Migrate or not takes the following logic:

  1. A Topologie Graph if built from the Locations (nodes) and Network Links (edges). Then the POPs and Clients are placed on this graph.
  2. At a certain period, data from the OpenStack Data Centers is gathered, aggregated and associated with the related POP. This is left to be scheduled on CronTab or a similar mechanism.
  3. A Gomory-Hu Tree is built from the Topologie Graph, based on the Network BW capacity. All the Demands are passed by the tree and consume some network capacity. At the end, the Gomory-Hu Tree has the remaining network capacity after all the Demands have been satisfied. For each Demand, a Shortest-path is found from the Client's Location to the demanded POP (the demanded POP holds an Instance of the demanded vCDN). The network links in this shortest-path have their capacity consumed to satisfy the Demand's BW. The Demands for which there is no valid Instace (the demanded POP does not contain the demanded vCDN) are declared invalid and ignored.
  4. After all the Demands have been analyzed, some of the network links will have no remaining link capacity, or would be saturated (negative remaining link capacity). Otherwise, if all links have remaining capacity, it means that the network can easily serve all the demands and there is no need to migrate.
  5. Every Demand is re-analyzed to verify if it traverses a saturated link. If it does, then a Migration is needed. If there is a POP in the Demand's path that can hold the demanded vCDN (Using the needed capacity values set for the vCDN and the current available resources from OpenStack metrics) and if the POP has the network capacity to hold the streaming (by calculating the difference between the installed BW and the currently used BW), then a Migration is possible and its cost and delay is calculated.
  6. However, if the Migration would take place, all the other Demands using the vCDN from the original POP would have to be redirected to the new POP. So, the Migration is only possible if there redirected Demands do not cause another link saturation. This is done by comparing the BW capacity recovered because of the Migration versus the BW capacity needed to accomodate all the Demands that have to be redirected because of the Migration.
  7. If the destination POP found for a Migration already has an Instance of the requested vCDN, then there is no need to Migrate but to Scale-out the destination POP and redirect the Demands to the optimal-located POP.
  8. In the end, by executing the recomended Operations, the Infrastructure will serve better the set of Demands. A subset of these recomendations can be simulated to verify its expected impact in the infrastructure.

Simulation


The Simulation shows how the Topologie Network BW Capacity is consumed if the selected changes were applied, affecting the Instances and adjusting the Demands to reflect the changes.
The Simulation of the Optimization follows the following logic:

  1. For the Selected Migrations, the Instace is migrated to the destination POP and all Demands using the migrated Instace are adjusted to demand the same vCDN in the destination POP. This might affect more than 1 Demand.
  2. For the Selected Redirections, the Demand is adjusted to request the same vCDN from the redirected POP. This might affect only 1 Demand, all other Demands are not affected.
  3. For the Selected Instantiations, the Instace is simulated to exist and all the Demands requesting the new Instance are declared valid and its BW is consumed from the Topologie Capacity.

A list of the simulated operations is shown and could be selected for actual execution in the OpenStack Data Centers. Additionally, a list of changes in the Demands is shown, which are needed for the Demands to adjust to the new optimized infrastructure (the Demand CDN direction mechanism is assumed external to vIOS and OpenStack systems).

Operations in the OpenStack Data Centers


Optimization

When declaring the POPs of the infrastructure, the OpenStack endpoint and credentials are given. Either manually or periodically, the system connects to the declared OpenStack Controller and uses Nova API calls to obtain information about the status of the Compute Nodes ( A POP is considered composed of one or more Compute Nodes).
When declaring the vCDNs, the OpenStack credentials are given. Either manually or periodically, the system connects to the declared POP OpenStack Controllers and uses the vCDN credentials as TENANT for the Nova API. Then the used and quota resources are queried for the vCDN TENANT (The TENANT does not have knoledge of the Compute Nodes, it sees all the available resources as a pool). Additionally, using the vCDN TENANT credentials, the systems calls the Ceilometer API to query for usage statistics of resources for the vCDN TENANT. These statistics are gathered and aggregated over a period of last X hours. In particular, these metrics are used to calculate the BW usage or the OpenStack Data Centers.

Execution

After the Simulation stage, a set of operations could be selected for implementation in the OpenStack Data Centers. These can be either Migrations or Instantiations.
The Migrations can be accomplished either by cloning the existing VMs via snapshots (Pause the VMs, take snapshots, download images, ppload images, create VMs, start VMs) or via metadata (Create VMs with exactly the same parameters as the original VMs). After the Migration, the original VMs are deleted; the floating ips returned to the pool.
For both approaches, it is needed that all the OpenStack Data Centers have a unified set of Flavors, Networks and Security Groups. In the second case, it is needed that all the OpenStack Data Centers have a unified set of Images available for instantiation.
If the original VM had a floating ip associated, the migrated VM is also asked to have a floating ip in the destination OpenStack Data Center. The floating ip can be completely different from the original one as each OpenStack DataCenter can have its own pool of public IP addresses and its own management and subneting policies.
If configured, the system will connect to a OpenStack Designate DNS Server to also move the DNS A Records from the old floating ip to the new one. If the vCDN has defined its DNS Domain, the Designate API will contact the Designate Server using the vCDN TENANT credentials. Otherwise, the default DNS Domain and admin TENANT credentials will be tried in the Designate API.
The Instantiations are accomplished by cloning an existing Instance or the vCDN, as vIOS and HMAC assume that all the Instances of a vCDN have exactly the same size and characteristics.

Note:

vIOS does not monitor the progress of any of the operations in the Web interface, the status of the operations is to be monitored directly from the OpenStack Controllers and from the messages and logs of vIOS. This is because the operations are executed by independent threads that do not report to the Web interface.

QoE BW coefficient


The Demands BW can be adjusted to satisfy a certain QoE value that the Infrastructure would be able to provide, in total average, to the Clients.
This allows the Operator to determine how much BW less or more is needed to assure a QoE value for the clients. This QoE value could be part of an SLA and is important for the Operator to have control over it.
The model NCUPM provides a BW coefficient based on the QoE parameter (measured in MOS values). This is an exponential model, whose parameters can be set in the INI config files of vIOS.

Optimization Frequency


The Migration is an expensive operation, so the migration frequency requieres attention.

  • Ideally there should be a syncronization among the frequency and time period of the OpenStack metrics and statistics gathering. Meaning that every X hours the OpenStack Telemetry service is asked to produce the statistics of the last Y hours . The values X hours and Y hours can be different between them.
  • The Demands, for optimization, are expected to be produced by statistical analysis. Meaning that the Demands represent the average/maximum/mean demand on the infrastructure for the last Z hours. In the case of CDNs; this statistics can be gathered by the DNS redirection system and aggregated in the manner that suits the business needs (average, maximum, etc). The data for this analysis and its aggregation is outside of the scope of vIOS, this is expected to be an input. The value Z of the analysis should be comparable with the OpenStack metrics querying frequency to make an adequate optimization decision (remember both are input for the optimization decision).
  • The Optimization operation should be executed in accordance with the data aggregation frequency of the Demands and the OpenStack values. Example: The Optimization is executed every 24 hours, based on Demands aggregated over the last 24 hours (or a prediction of the Demand in the next 24 hours), with the OpenStack operational capacity of the last 48 hours (updated every 6 hours).

This version of vIOSimulator is designed and adjusted for the DVD2C project, on which Telecom SudParis has a participating role.
